Skills · Procedural memory, on demand

Small workflow files that teach the agent how to work without stuffing every rule into the base prompt.

Prompt size is a design constraint

It is easy to solve early agent behavior problems by adding more instructions to the system prompt. That works until the prompt becomes a junk drawer. The model sees too much irrelevant procedure, important rules compete for attention, and every new capability increases the chance of accidental conflict.

The skills system is my answer to that. A skill is a small Markdown procedure with a description that says when it applies. The agent can load a skill for a GitHub review, a research sweep, a media generation workflow, or a deployment checklist without carrying all of those procedures into every conversation.

What a skill contains

A useful skill is not a book. It names the trigger, the inputs, the sequence, and the verification step. The most valuable skills are operational: they encode how to inspect first, how to avoid destructive actions, how to verify outcomes, and how to report what changed.

I keep the categories broad and non-sensitive: creative, devops, github, media, mlops, productivity, research, software-development, and social-media. The point is not the category list; it is the habit of moving repeated procedure into a readable artifact that can be improved over time.

Illustrative skill loader
# illustrative
def choose_skills(task, catalog):
    matches = []
    for skill in catalog:
        if skill.applies_to(task):
            matches.append(skill.summary())
    return sorted(matches, key=lambda item: item.confidence, reverse=True)[:3]

Trade-offs

The main risk is overfitting procedure. A skill should guide the model, not trap it into a brittle script. I write skills as checklists and principles rather than exact transcripts. That leaves room for judgment while preserving the parts I do want repeated: inspect before editing, avoid secrets, verify with tests, and summarize clearly.

Another risk is stale procedure. A skill that references old commands or outdated APIs can make the system worse. Treating skills as source-controlled artifacts helps because changes are visible and reviewable.

Skill selection also needs restraint. If the agent loads five procedures for a simple request, the extra context can make the answer worse. I prefer a small catalog match with clear reasons: this task is a GitHub review, this task is media generation, this task is research synthesis. The skill should make the next action more obvious.

In practice, skills have become a way to preserve operational taste. They encode habits like reading the existing code before editing, using focused tests, avoiding unrelated refactors, and explaining residual risk. Those habits are hard to express as one giant universal rule, but they fit naturally as small procedures tied to concrete work.

What I learned

Procedural memory is one of the highest-leverage parts of an agent platform. It converts lessons into reusable behavior without making the core prompt enormous. The best skills read like notes from an experienced teammate: short, specific, and biased toward verification.

I also learned that a skill is easier to maintain when it is honest about its limits. Some tasks need current documentation, some need human approval, and some should stop if the workspace is dirty. Writing those constraints directly into the procedure makes the agent more dependable.

The discipline is similar to writing runbooks: make the common path clear, name the edge cases, and keep the document short enough that it will actually be read during the task.
